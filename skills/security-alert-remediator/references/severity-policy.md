# 严重级别聚焦策略

## 目标

这个技能不追求“一次处理完所有告警”，而是优先清掉当前最危险、最可修复的安全项。

## 动态最低级别

默认行为不是固定死在某个 severity，而是根据当前告警分布动态下探：

| 当前告警分布 | 本轮聚焦级别 | 说明 |
| --- | --- | --- |
| 存在 critical，且 high+ 总量达到高压阈值 | critical | 先清最危险的一层，避免在 high 噪音里分散精力 |
| 存在 high 或 critical，但数量未达到高压阈值 | high | 默认忽略 medium 及以下 |
| 不存在 high+，但存在 medium | medium | 只在没有更高等级时处理 medium |
| 只有 low / note / warning | low | 作为最后一轮收尾 |

默认高压阈值为 5 个 high+ 告警。脚本允许通过参数覆盖。

## 同级排序规则

同一 severity 内，按下面顺序处理：

1. 有明确 patched version 的 Dependabot 告警
2. 同时影响多个 manifest 或出现次数更多的依赖
3. 需要少量代码修复即可落地的 Code Scanning 告警
4. 没有补丁版本、只能暂时观察或人工评估的告警

## 特殊处理

- Code Scanning 告警默认不自动修复，除非定位清晰且修复范围很小。
- 对于 high+ 但没有补丁版本的依赖，必须进入单独遗留清单。
- 如果同一包触发多个不同 advisory，优先一次性判断该包的目标版本，而不是反复震荡升级。
- 如果多个包存在 peer 或发布顺序约束，先确认依赖关系，再决定是否成组升级。

## 输出要求

告警采集阶段至少要输出以下信息：

- 当前 focusSeverity
- 各 severity 数量统计
- 可直接升级的包候选列表
- 需要人工代码修复的 Code Scanning 清单
- 无法直接升级但仍属于 high+ 的依赖清单
