# 安全告警修复手册

本文中出现的 scripts/ 路径都相对于当前技能目录。如果技能是全局安装的，先定位当前技能目录，再解析 ./scripts/*；不要假设这些脚本存在于用户项目根目录。

## 1. 仓库前置条件

- 工作区必须干净。任何未提交改动都会污染升级结果，也会让回滚和分批提交失去边界。
- 优先用 fast-forward 同步远端。如果本地与远端已分叉，不要直接自动 merge。
- 如果冲突只发生在纯生成文件，且可以确定为机械冲突，才允许尝试修复；否则中断。

## 2. 批次策略

默认一次只更新一个依赖。以下情况才允许同时更新多个依赖：

- 明确成对出现的核心包，例如 react 与 react-dom
- 同一生态要求同步版本的工具链包
- 一个依赖的修复版本要求配套更新 peer dependency

如果多包批次失败：

1. 记录失败批次和错误。
2. 回退到单包升级策略。
3. 重新验证每个单包是否可以独立落地。

## 3. 验证等级

### 最小验证

适用于单包 patch/minor 升级且无代码改动：

- lint
- test 或 typecheck 中至少一个真实存在的快速检查

### 完整验证

适用于多包升级、major 升级、修复 lockfile 之后、或伴随代码改动时：

- lint
- test（如果仓库存在）
- build
- typecheck（如果仓库存在）

## 4. 破坏性变更判定

可以接受：

- 几行兼容适配
- import 路径或 API 名称的小范围调整
- 锁文件重建、peer 版本对齐、测试基线小修正

不应自动接受：

- 需要跨模块重构
- 需要改动业务逻辑或协议层
- 需要引入新的基础设施、配置体系或大面积测试重写

遇到不可接受的破坏性更新时：

1. 停止该包继续升级。
2. 保留已经成功通过验证的前置修复成果。
3. 把该包作为阻塞项单独反馈给用户。

## 5. Override 清理策略（显式启用）

- override 清理默认关闭。只有用户明确要求评估或移除某个 pnpm override 时，才允许进入这条分支。
- override 的定位是临时缓解措施，不是永久依赖治理方案；一旦怀疑它已经失效，可以评估是否移除，但验证强度必须高于新增 override。
- 默认一次只移除一个 override selector。只有多个 selector 明确汇聚到同一个叶子依赖，且用户同意按包名成组处理时，才允许一起移除。
- 精确 selector 优先于模糊匹配。只有确实存在同一叶子依赖的多条 selector 时，才使用按包名清理模式。

### Override 清理的最小步骤

1. 先用 <skill-dir>/scripts/collect-security-alerts.mjs 记录移除前快照。
2. 运行 <skill-dir>/scripts/remove-pnpm-override.mjs，默认保留安装、frozen 校验和告警对比。
3. 确认移除后没有重新引入目标包依赖告警，也没有新增达到阈值的依赖告警。
4. 继续执行完整质量门，而不是停在 lockfile 校验通过这一层。

### Override 清理的失败处理

- 如果移除后重新出现目标包告警，立即视为失败，不要保留这次清理。
- 如果移除后出现新的 high+ 依赖告警，也视为失败，除非用户明确接受人工继续调查。
- 脚本回滚后，仍然要把失败原因写进汇报，不要让 override 清理失败被悄悄吞掉。

## 6. 提交策略

- 如果中途出现难以接受的破坏性更新，先提交前面已经验证通过的安全修复。
- 如果所有可修复项都已经通过完整验证，可以合并成一次提交；否则按批次拆分提交。
- 任何包含代码修复的依赖升级，都应该先经过最小 review，再决定是否进入提交。

## 7. 收尾汇报

汇报至少包含：

- 已修复的告警和对应依赖版本变化
- 每个批次执行的质量门
- 若启用了 override 清理：被移除的 selector、是否回滚、前后告警差异
- 是否出现 code scanning 代码修复
- 未解决的 high+ 依赖告警
- 需要人工继续处理的阻塞项
