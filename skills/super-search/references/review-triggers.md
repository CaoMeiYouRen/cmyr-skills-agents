# Review Triggers

## 自动触发条件

### 1. 高风险主题（强制触发）

当搜索主题涉及以下领域时，**自动触发**对抗性审查：

| 领域 | 关键词（中英） | 审查重点 |
|------|---------------|----------|
| **医学/健康** | 医学、医疗、药物、疾病、治疗、手术、临床、药效、副作用、health、medical、drug、treatment、disease | 药品疗效声明是否被权威机构认可；是否有临床数据支撑；是否遗漏禁忌/副作用信息 |
| **法律** | 法律、诉讼、合同、判决、法规、违法、犯罪、律师、法院、legal、law、court、contract | 法律条文引用是否准确；判例是否真实；是否遗漏管辖/时效关键信息 |
| **金融/投资** | 金融、股票、投资、基金、理财、保险、贷款、利率、finance、stock、investment | 数据来源是否可靠；是否存在误导性收益承诺；是否遗漏风险提示 |
| **安全** | 安全、漏洞、攻击、黑客、加密、认证、security、vulnerability、hack、exploit | 漏洞信息是否经过官方确认；修复建议是否正确；是否遗漏影响范围 |
| **食品安全/营养** | 食品安全、营养、保健品、膳食、添加剂、food safety、nutrition | 健康声称是否有科学依据；是否引用权威机构意见 |

| **开源/GitHub 项目** | github、开源项目、open source、⭐、repo、仓库、GitHub Trending、star、fork | 星数真实性（抓取 API 确认）、项目活跃度（last commit/release 日期）、Issues 健康度（open issues 数量和响应速度）、许可证风险（AGPLv3/GPLv3 等强传染性许可）、第三方转述准确性（必须抓取原始 README 交叉对照） |

### 1.5 开源/GitHub 项目核查 ⚠️ 高频场景

当搜索主题涉及开源项目选型、对比或推荐时，**必须执行以下核查**：

**强制核查项**：

| 核查项 | 风险 | 方法 |
|-------|------|------|
| ⭐ 星数真实性 | 聚合文章可能引用过期/错误星数 | 抓取 GitHub API 或 repo 页面确认 |
| 📅 活跃度 | 看起来高星但已停更 | 检查 last commit 和 latest release 日期 |
| 🐛 Issues 健康度 | 项目 bug 多、维护者不响应 | **强制**：扫描 open issues 前 2 页，关注 bug 标签、stale PR、维护者最后回复时间 |
| 📜 许可证 | AGPLv3 等强传染性许可影响商用 | 读取 LICENSE 文件 |
| 🔗 描述准确性 | 第三方文章中的项目描述可能夸大/过时 | 抓取原始 README 交叉对照 |

**判定标准**：
- 项目出现在聚合推荐文章中 → 必须追溯原始 repo
- 报告中引用了 star 数 → 必须标注数据获取日期（star 数会变化）
- 涉及「推荐」「选型」「对比」→ 必须检查许可证兼容性
- 项目最后 commit 超过 12 个月 → 必须标注「⚠️ 可能已停更」

### 2. 科学主题（原理核查）

当搜索主题涉及以下科学领域时，**从基础原理出发核查**：

| 领域 | 关键词 | 审查方法 |
|------|--------|----------|
| **物理学** | 物理、力学、量子、相对论、电磁、热力学、physics | 检查是否符合基本物理定律（能量守恒、热力学定律等） |
| **化学** | 化学、分子、元素、反应、化合物、催化、chemistry | 检查化学反应方程式是否平衡；是否存在常识性化学错误 |
| **数学** | 数学、定理、公式、证明、算法复杂度、math | 检查公式是否正确；定理引用是否存在谬误 |
| **生物学** | 生物、基因、细胞、进化、DNA、RNA、biology | 检查是否符合进化论/分子生物学基本原理 |

审查步骤：
1. 提取所有科学声明
2. 逐一与已知基础原理对照
3. 对存疑声明搜索官方/学术渠道验证
4. 标注"未经验证"的声明

### 2.5 第三方聚合报告陷阱 ⚠️ 必须检查

当信息来源为**第三方聚合/研究报告**（如 smartcity.team、行业白皮书、调研机构合集）时，报告中引用的**案例数据和数值声明**本质上是二次转述，**不可直接采信为事实**。

判定标准：
- 报告引用了一个具名人物/公司的案例，但**未提供原始链接或采访记录**
- 数值声明（收入、用户数、增长率）仅出现在聚合报告中，搜不到一手来源
- 报告本身标注为"研究/分析/洞察"而非一手调查

处理规则：
1. 对聚合报告中的每个案例数值声明，**必须追溯一手来源**
2. 搜不到一手来源的 → 标注"第三方转述 ⚠️，置信度：中等"或直接删除
3. 一手来源与聚合报告数据一致的 → 优先引用一手来源，聚合报告作为佐证
4. **绝不**将聚合报告中的二手数据当作"已验证事实"写入最终报告

### 3. 交叉验证矛盾或孤立触发

当 Step 6 交叉验证发现以下情况时**自动触发事实核查**：
- 关键发现出现**矛盾**（不同来源对同一事实给出不同数据）
- 关键发现为**孤立声明**（只有一个源提到，且该源非官方）
- 任一数值/规格类信息仅来自第三方文章，未经验证
- ⚠️ 数值声明来自第三方聚合报告（参见上方 2.5 节）

核查步骤：
1. 直接 fetch 涉及的官方页面（定价页、规格页、文档页）
2. 以官方第一手数据为终极裁决依据
3. 多个第三方来源的一致意见**不能**覆盖官方页面的明文数据
4. 官方页面无法访问时，标注为"未核实"并降低置信度
5. 将核查结论写入报告交叉验证章节

### 4. 手动触发

用户可通过以下方式手动触发：
- 明确要求："帮我审查""验证这个结论""有没有遗漏"
- 传入 `--force-review` 参数
- 设置环境变量 `SUPER_SEARCH_FORCE_REVIEW=1`

## 审查执行流程

1. **生成反驳搜索词**：对每个主要结论，生成对立面/批评性搜索词
2. **多样性检查**：检查来源是否覆盖至少 2 个不同立场
3. **补充搜索**：执行反驳性搜索，抓取对立观点
4. **对比分析**：比对正反方论据，标注置信度
5. **输出审查报告**：列出审查发现、遗漏风险和补充建议
