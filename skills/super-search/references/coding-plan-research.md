# AI Coding Plan / 订阅套餐定价与性价比调研工作流

用于深度调研一个 AI 编程订阅套餐（Coding Plan）的定价、额度、模型包含、使用限制以及竞品对比。适合 `depth=normal` 模式。

---

## 与 AI 模型调研的区别

| 维度 | AI 模型调研 | Coding Plan 调研 |
|:-----|:-----------|:----------------|
| **核心问题** | 模型能力、技术架构、benchmark | 价格、额度、模型覆盖、限制条件 |
| **关键信源** | 官方博客、arXiv 论文、HuggingFace | **官方定价页、社区对比页（CodingPlan.org）、Reddit 投诉帖** |
| **数据敏感点** | Benchmark 分数、参数量 | **价格区域差异（US vs CNY）、额度倍率、隐藏限流** |
| **对抗搜索重点** | 社区对模型能力的质疑 | **服务限流/429 投诉、涨价史、额度未兑现** |
| **主要输出** | 模型能力定位、使用推荐 | 套餐横向对比表、性价比评级、避坑提醒 |

## 关键分析维度

### 1. 套餐定价（需要多源确认）

| 维度 | 说明 |
|:-----|:------|
| **月费（美元）** | 官方定价，注意有会海内外差异 |
| **月费（人民币）** | 国内版价格，通常按¥8左右汇率兑换但可能不同 |
| **年费折扣** | 年付通常有折扣，但**不推荐年付**（政策变动风险大）|
| **首月优惠** | 很多套餐有首月折扣或免费试用 |

### 2. 额度体系（核心！）

| 维度 | 说明 |
|:-----|:------|
| **额度单位** | prompts次数 / token量 / 美元价值（三种不同计费方式） |
| **滚动窗口** | 5小时滚动窗口 vs 固定时间段重置 |
| **周/月上限** | 是否有周限额或月总额度 |
| **高峰期差异** | 部分套餐（如GLM）高峰期消耗倍数更高（2x-3x） |
| **并发限制** | 同时可用终端数是否受限 |

### 3. 额度倍率（性价比关键指标）

**公式**：额度倍率 = 当前周期额度价值 ÷ 月费价格

| 倍率 | 含义 | 例子 |
|:----:|:-----|:-----|
| <1 | 性价比差，月费超过实际获得价值 | GLM Lite 0.24 |
| 1-5 | 一般 | GLM Pro ~3.9 |
| 5-10 | 较好 | OpenCode Go ~6x, Ollama Pro ~10x |
| 10+ | 极好 | MiniMax Plus ~22x |

> ⚠️ 额度倍率是一个相对指标，前提是认可该模型的单位定价。高倍率不一定代表总价值更高。

### 4. 模型覆盖

| 维度 | 说明 |
|:-----|:------|
| **模型数量** | 独家模型 vs 多厂商模型聚合 |
| **模型能力** | SWE-bench/其他基准分数 |
| **是否含闭源旗舰** | 是否包含 Claude Opus、GPT-5.5 等 |
| **场景限制** | 是否只能在特定工具中使用 API Key |

### 5. 服务品质

| 维度 | 说明 |
|:-----|:------|
| **限流/429情况** | Reddit 等社区的真实体验 |
| **速度/延迟** | 用户反馈的 TPS（每秒 token 数） |
| **算力充裕度** | 高峰期是否频繁 429 |
| **涨价历史** | 是否曾突然涨价/降额度 |

## 数据交叉验证要点

| 数据类别 | 验证方法 |
|---------|:---------|
| **官方定价** | 官方页 + 至少 1 个第三方对比站确认 |
| **额度限制** | 官方文档为主，awesome-coding-plan 实测值为辅 |
| **额度倍率** | 由 awesome-coding-plan 作者计算，可引用的量化指标 |
| **社区投诉** | 需要多个独立用户的一致投诉才可采信 |
| **区域价格** | US 版和国内版 CNY 通常不同，必须分开标注 |
| **涨价历史** | 社区帖子 + 什么值得买/类似平台交叉确认 |

### 处理矛盾的原则

1. 定价矛盾 → 以**官方定价页**为准（页面截图或爬取内容）
2. 额度矛盾 → 以官方文档白纸黑字为准，社区实测作为参考
3. 服务稳定性矛盾 → 以**多个独立用户的共识**为准。有人快有人慢是正常的；所有人都说慢才可信
4. API Key 场景限制矛盾 → 以 **官方文档**为准，不要采信第三方文章中过时的信息

## 报告格式要点

### 表格设计原则

1. **维度对齐**：所有对比套餐使用相同维度列（价格、模型数、5h额度、额度倍率、场景限制）
2. **价格带分组**：按 `$0-5 / $10 / $15-20 / $100+` 分层呈现
3. **高亮最佳值**：在对比表中用 **加粗** 标记每个维度的最佳值
4. **标注来源**：所有定价单元格标注来源编号

### 关键锚点指标

| 指标 | 解释 |
|:-----|:------|
| **额度倍率** | 你付的钱能买到多少"额度价值"，越高越划算 |
| **5小时限额** | 滚动窗口内的使用上限，决定爆发式使用的可行性 |
| **模型覆盖** | 能用的模型数量和质量 |
| **服务稳定性** | 是否频繁限流/429 |

### ⚠️ 警告标注模板

```
> ⚠️ **{品牌} 高峰期（{时间}）消耗 {N}x 配额**：这意味着高峰期实际可用额度仅为标称的 1/{N}。
> ⚠️ **社区广泛反馈限流/429 问题**：参见 [来源](link)。
> ⚠️ **涨价历史**：202X年X月曾涨价 ≥{N}%，同时额度降低 {N}%。
```

## 常见坑

- ❌ 只搜索官方定价页而不检查社区真实反馈（限流问题常被官方隐瞒）
- ❌ 把 US 定价和国内 CNY 定价混为一谈（不同版本可能定价和额度不同）
- ❌ 忽略高峰期差异（如 GLM 14-18 点 3x 消耗）
- ❌ 只比较月费不比较"额度倍率"（$3 但只能调用 80 次/5h ≠ 划算）
- ❌ 不关注涨价历史（AI 套餐行业普遍是"先低价获客，再涨价收割"）
- ❌ 只搜正面评价不做对抗性搜索（看 Reddit 最差评论比好评更有价值）
- ❌ 忽略额度计算方式差异（prompts 次 vs token 量 vs 美元价值，不可直接比较）
- ❌ 将第三方转载的价格表当作一手数据（转载可能导致信息过时）
